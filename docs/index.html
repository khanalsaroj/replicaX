Features

Everything to clone a workflow,
nothing that shouldn't travel.

One-command capture

init scans the current project into a reusable profile under .replicax/.

Capture any GitHub repo

extract owner/repo profiles a remote repo — no clone, no git binary required.

One-command scaffold

create reproduces the exact setup in a fresh directory — same tooling, zero code.

AI assistant skills

init-skill uses your own AI (Claude / Codex / Gemini) to author a ready-to-use skill for agentic coding tools.

.ts and .js configs

Copied byte-for-byte — never compiled, never executed. Both variants just work.

Secret-safe by design

.env, keys, and certs can never enter a profile; .npmrc tokens are stripped automatically.

Stays in sync

sync --diff shows what drifted; validate verifies integrity via SHA-256.

Portable & shareable

export / import a whole profile as a single, validated .tar.gz.

Precise control

.replicaxignore excludes, .replicaxinclude adds — gitignore-style control over exactly what travels.

Quick start

Two commands. That's the loop.

Capture a well-configured project once, then scaffold as many fresh ones as you like.

01

Capture an existing project

In a project you've already set up the way you like:

$ cd my-project && replicax init

→ writes a profile to .replicax/. Add --dry-run to preview without writing.

02

Scaffold a fresh one

Anywhere the profile lives, reproduce the setup:

$ replicax create my-new-app

→ same tooling and structure, none of the code. Or pull from a remote repo with replicax extract owner/repo.

Security

Secret exclusion is a hard guarantee,
not a best effort.

These boundaries are enforced in code and covered by tests — not configuration you can accidentally turn off.

Secrets are never captured

.env, .env.*, *.pem, *.key, *.crt, SSH keys and friends are blocked unconditionally — this cannot be overridden by configuration.

.npmrc is sanitized

A legitimate setup file — but auth tokens (_authToken, _password, …) are stripped out before it ever enters a profile.

No path escapes

Every path from a profile or AI response is validated against traversal (..) and absolute paths before anything is written. validate re-checks it.

Archive extraction is a trust boundary

An imported .tar.gz is validated before a byte is written: traversal, symlinks/hardlinks and device entries are rejected, with caps on size, file count, and per-file size — tar bombs aborted up front.

Install is opt-in for untrusted profiles

Because installing runs lifecycle scripts, create won't auto-install for an extracted or imported profile — it prints the dependency list and waits for an explicit --install.

No business source is captured

Only config files and the empty folder hierarchy enter a profile. Application source is never read; runtime dependencies are dropped from package.json.

How it works

Scan → guard → profile → reproduce.

A two-pipeline design: capture filters everything through a non-overridable secret guard, and reproduce validates every path on the way out.

01

Scanner

Detects config files, the folder hierarchy, and metadata — locally or from a GitHub tarball.

02

Ignore + secret guard

Layers default ignores and .replicaxignore over a separate, non-overridable secret guard.

03

Profile generator

Assembles the bundle of 5 JSON files and computes SHA-256 checksums for integrity.

04

Project generator

Loads + zod-validates the bundle, then reproduces the setup with a conflict resolver for existing files.

FAQ

Good questions, short answers.

Does it copy my source code?
No. Only configuration files and the empty folder hierarchy. Application code is never read into a profile.
What about .ts configs like vite.config.ts?
Fully supported — they're copied as text, so both .ts and .js variants work without any compile step.
Can a profile leak a secret?
No. Secret exclusion is unconditional and .npmrc tokens are stripped; validate fails if a secret ever appears in a profile.
Why are runtime dependencies dropped from package.json?
They're part of your application, not your setup. devDependencies, scripts, and engines are kept; the new project's name is stamped in on create.
Does extract need git installed?
No. It downloads the repo as a tarball over the GitHub API using Node's built-in fetch — no git binary, no full clone.
Is it cross-platform?
Yes — Windows (native + WSL), macOS, and Linux.
