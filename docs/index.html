Capabilities Fig. 01

What travels

Everything needed to clone a workflow — and nothing that shouldn't leave the building.

01

One-command capture

init scans the current project into a reusable profile under .replicax/.

02

Any GitHub repo

extract owner/repo profiles a remote repo — no clone, no git binary required.

03

One-command scaffold

create reproduces the exact setup in a fresh directory — same tooling, zero code.

04

AI assistant skills

05

.ts and .js configs

Copied byte-for-byte — never compiled, never executed. Both variants simply work.

06

Secret-safe by design

.env, keys, and certs can never enter a profile; .npmrc tokens are stripped automatically.

07

Stays in sync

sync --diff shows what drifted; validate verifies integrity via SHA-256.

08

Portable & shareable

export / import a whole profile as a single, validated .tar.gz.

09

Precise control

.replicaxignore excludes, .replicaxinclude adds — gitignore-style control over what travels.

Process Fig. 02

The method

A two-pipeline design: capture filters everything through a non-overridable secret guard, and reproduce validates every path on the way out.

Scan

Surveys config files, the folder hierarchy, and metadata — locally or from a GitHub tarball.

Guard

A non-overridable secret guard sits beneath default ignores and your .replicaxignore.

Reproduce

Loads and zod-validates the bundle, then rebuilds the setup — every path checked on the way out.

01

Capture a project

In a project you've already set up the way you like:

$ cd my-project && replicax init

→ writes a profile to .replicax/. Add --dry-run to preview without writing.

02

Scaffold a fresh one

Anywhere the profile lives, reproduce the setup:

$ replicax create my-new-app

→ same tooling and structure, none of the code. Or pull from a repo with extract owner/repo.

Boundaries Fig. 06

A hard guarantee

Secret exclusion is enforced in code and covered by tests — not configuration you can accidentally turn off.

01

Secrets are never captured

.env, *.pem, *.key, *.crt, SSH keys and friends are blocked unconditionally — config cannot override it.

02

.npmrc is sanitized

A legitimate setup file — but auth tokens (_authToken, _password, …) are stripped before it ever enters a profile.

03

No path escapes

Every path from a profile or AI response is validated against traversal (..) and absolute paths before anything is written.

04

Archives are a trust boundary

An imported .tar.gz is validated first: traversal, symlinks and device entries are rejected, with caps on size and file count.

05

Install is opt-in

Because installing runs lifecycle scripts, create won't auto-install an untrusted profile — it waits for an explicit --install.

06

No business source

Only config and the empty folder hierarchy enter a profile. Runtime dependencies are dropped from package.json.

Questions Fig. 07

Short answers

Does it copy my source code?
No. Only configuration files and the empty folder hierarchy. Application code is never read into a profile.
What about .ts configs like vite.config.ts?
Fully supported — they're copied as text, so both .ts and .js variants work without any compile step.
Can a profile leak a secret?
No. Secret exclusion is unconditional and .npmrc tokens are stripped; validate fails if a secret ever appears in a profile.
Why are runtime dependencies dropped?
They're part of your application, not your setup. devDependencies, scripts, and engines are kept; the new project's name is stamped in on create.
Does extract need git installed?
No. It downloads the repo as a tarball over the GitHub API using Node's built-in fetch — no git binary, no full clone.
Is it cross-platform?
Yes — Windows (native + WSL), macOS, and Linux.
