Capture once. Reproduce anywhere.

One profile holds a project's tooling and folder structure. Never its source, never its secrets. Two commands cover the whole loop.

Capture a project

In a project you have already set up the way you like:

$ replicax init

Writes a profile to .replicax/. Add --dry-run to preview without writing.

Scaffold a fresh one

Anywhere the profile lives, reproduce the setup:

$ replicax create my-new-app

Same tooling and structure, none of the code. Or pull from a repo with extract owner/repo.

Capabilities

What travels

Everything needed to clone a workflow, and nothing that should stay behind.

One-command capture

init scans the current project into a reusable profile under .replicax/, ready to share or reuse.

$ replicax init

Any GitHub repo

extract owner/repo profiles a remote repo. No clone, no git binary.

One-command scaffold

create rebuilds the exact setup in a fresh directory.

AI assistant skills

init-skill uses your own Claude, Codex, or Gemini to author a skill.

.ts and .js configs

Copied byte for byte. Never compiled, never executed.

Secret-safe by design

.env, keys, and certs can never enter a profile. .npmrc tokens are stripped automatically.

Stays in sync

sync --diff shows what drifted. validate verifies integrity via SHA-256.

Portable and shareable

export and import a whole profile as one validated .tar.gz.

Precise control

.replicaxignore excludes, .replicaxinclude adds. Gitignore-style control over exactly what travels.

Two pipelines, one guard

Capture filters everything through a non-overridable secret guard. Reproduce validates every path on the way out.

Scan

Survey the setup

Reads config files, the folder hierarchy, and metadata, locally or from a GitHub tarball.

Guard

Block secrets

A non-overridable secret guard sits beneath default ignores and your .replicaxignore.

Reproduce

Rebuild safely

Loads and zod-validates the bundle, then rebuilds the setup. Every path is checked on the way out.

Boundaries

A hard guarantee

Secret exclusion is enforced in code and covered by tests, not configuration you can accidentally turn off.

01

Secrets are never captured

.env, *.pem, *.key, *.crt, and SSH keys are blocked unconditionally. Config cannot override it.

02

.npmrc is sanitized

A legitimate setup file, but auth tokens (_authToken, _password) are stripped before it enters a profile.

03

No path escapes

Every path from a profile or AI response is validated against traversal (..) and absolute paths before anything is written.

04

Archives are a trust boundary

An imported .tar.gz is validated first. Traversal, symlinks, and device entries are rejected, with caps on size and file count.

05

Install is opt-in

Because installing runs lifecycle scripts, create waits for an explicit --install before touching an untrusted profile.

06

No business source

Only config and the empty folder hierarchy enter a profile. Runtime dependencies are dropped from package.json.

Short answers

Does it copy my source code?
No. Only configuration files and the empty folder hierarchy. Application code is never read into a profile.
What about .ts configs like vite.config.ts?
Fully supported. They are copied as text, so both .ts and .js variants work without any compile step.
Can a profile leak a secret?
No. Secret exclusion is unconditional and .npmrc tokens are stripped. validate fails if a secret ever appears in a profile.
Why are runtime dependencies dropped?
They are part of your application, not your setup. devDependencies, scripts, and engines are kept, and the new project's name is stamped in on create.
Does extract need git installed?
No. It downloads the repo as a tarball over the GitHub API using Node's built-in fetch. No git binary, no full clone.
Is it cross-platform?
Yes. Windows (native and WSL), macOS, and Linux.
